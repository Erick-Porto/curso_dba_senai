Boas Práticas e Padrões Profissionais (Aplicados ao Projeto)

Ao longo do projeto, aplicamos (ou deveríamos aplicar) várias boas práticas:

  • Nomenclatura Padronizada:
    • Adotamos PascalCase para nomes de Tabelas e Colunas no Modelo Conceitual/Lógico (ex: NomeProduto, DataCadastro).
    • Para scripts SQL (MySQL/PostgreSQL), a convenção comum e recomendada é snake_case para nomes de tabelas e colunas (ex: nome_produto, data_cadastro, livro_autores). Seus scripts CREATE TABLE devem refletir isso.
    • Chaves Primárias: nome_da_tabela_id ou apenas id (se único e claro no contexto da tabela).
    • Chaves Estrangeiras: nome_tabela_pai_id (ex: editora_id na tabela livros).
  • Comentários nos Scripts SQL:
    • Usar -- para comentários de linha.
    • Usar /* ... */ para blocos de comentários.
    • Comentar o propósito de tabelas, colunas com lógica não óbvia, constraints complexas, e seções do script.
    
-- Tabela para armazenar os detalhes dos autores dos livros
CREATE TABLE autores (
    autor_id INT AUTO_INCREMENT PRIMARY KEY,
    nome_autor VARCHAR(150) NOT NULL -- Nome completo do autor
    /* Adicionar mais campos como data_nascimento, nacionalidade, etc. */
);
  • Organização Modular (Para Projetos Maiores):
    • Em sistemas grandes, os scripts DDL (CREATE) são frequentemente separados em arquivos por módulo funcional (ex: schema_financeiro.sql, schema_rh.sql) ou por tipo de objeto (tabelas.sql, views.sql, indices.sql) para facilitar a manutenção e o versionamento.
  • Segurança e Validação na Fonte (Banco de Dados):
    • Maximizar o uso de constraints (NOT NULL, UNIQUE, CHECK, FOREIGN KEY com ações referenciais apropriadas) para garantir a integridade e validade dos dados diretamente no banco. Isso reduz a carga na camada de aplicação e previne dados ruins.
