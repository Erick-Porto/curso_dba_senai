MER - Refinando o Modelo

Alguns pontos para refinar nosso DER inicial:

  • Atributos Multivalorados:
    • Exemplo: Se um livro pudesse ter múltiplas "palavras-chave" diretamente como atributo (Livro: {..., PalavrasChave: "fantasia, aventura, medieval"}). Isso violaria a 1FN.
    • Solução Correta (1FN): Criar uma nova entidade (ex: PALAVRACHAVE) e um relacionamento N:M com LIVRO, resultando em uma tabela associativa (LIVRO_PALAVRASCHAVE).
    • No nosso caso, GENERO foi corretamente modelado como uma entidade separada com relacionamento N:M com LIVRO, o que já resolve isso para os gêneros.
  • Especialização/Generalização (Opcional para este projeto):
    • Se tivéssemos atributos comuns e específicos para diferentes tipos de "Pessoas" no sistema (ex: Autores, Membros, Funcionários da Biblioteca).
    • Poderíamos ter uma superclasse PESSOA (com Nome, Documento) e subclasses AUTOR (com Biografia), MEMBRO (com DataValidadeCartao).
    • Prós: Reduz redundância de atributos comuns, representa melhor a hierarquia conceitual. Contras: Adiciona complexidade ao modelo e às consultas (requer mapeamento específico para tabelas).
    • Para o nosso projeto de biblioteca, manter AUTOR e MEMBRO como entidades separadas é provavelmente mais simples e adequado ao escopo, a menos que haja muitos atributos significativamente em comum.
