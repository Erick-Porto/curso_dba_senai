Boas Práticas e Padrões Profissionais (Aplicados ao Projeto)

Ao longo do projeto, aplicamos (ou deveríamos aplicar) várias boas práticas:

  • Nomenclatura Padronizada:
    • Usamos CamelCase ou PascalCase para tabelas e colunas no Modelo Conceitual/Lógico (ex: NomeProduto, DataCadastro).
    • No script SQL (MySQL/PostgreSQL), é comum e recomendado usar snake_case para nomes de tabelas e colunas (ex: nome_produto, data_cadastro, livro_autores). (Adaptar o script se necessário para refletir isso).
    • Chaves Primárias: NomeDaTabelaID ou apenas ID.
    • Chaves Estrangeiras: NomeDaTabelaPaiID.
  • Comentários nos Scripts SQL:
    • Usar -- para comentários de linha e /* ... */ para blocos.
    • Comentar o propósito de tabelas complexas, colunas com lógica específica, ou constraints não óbvias.
    -- Tabela para armazenar os exemplares físicos dos livros
CREATE TABLE Exemplares ( ... );
  • Organização Modular (Para Projetos Maiores):
    • Em sistemas grandes, os scripts DDL podem ser divididos em arquivos por módulo funcional ou por tipo de objeto (tabelas, views, índices, etc.) para facilitar a manutenção.
  • Segurança e Validação na Fonte:
    • Maximizar o uso de constraints (NOT NULL, UNIQUE, CHECK, FOREIGN KEY) para garantir a integridade dos dados diretamente no banco, reduzindo a carga na aplicação.
